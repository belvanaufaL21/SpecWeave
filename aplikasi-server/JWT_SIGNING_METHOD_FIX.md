# JWT Signing Method Fix

## Problem Identified
Backend authentication was failing with this specific error:
```
⚠️ [AUTH] Token verification error: invalid JWT: unable to parse or verify signature, token signature is invallid: signing method ES256 is invalid
```

## Root Cause Analysis

### JWT Signing Methods in Supabase

**User Access Tokens (ES256)**:
- Generated by Supabase Auth for user sessions
- Signed with **ES256** (Elliptic Curve Digital Signature Algorithm)
- Used by frontend applications
- Short-lived (typically 1 hour)

**Service Role Tokens (HS256)**:
- Used for server-to-server communication
- Signed with **HS256** (HMAC with SHA-256)
- Used for admin operations
- Long-lived or permanent

### The Problem

Backend was using **service role client** to verify **user access tokens**:

```javascript
// WRONG: Service role client cannot verify user tokens
const supabaseAdmin = createClient(supabaseUrl, supabaseServiceKey);
const { data: { user }, error } = await supabaseAdmin.auth.getUser(userToken);
// Error: ES256 signing method is invalid for service role client
```

Service role clients expect HS256-signed tokens, but user tokens are ES256-signed.

## Solution Implemented

### 1. **Use Anonymous Client for User Token Verification**

```javascript
// CORRECT: Anonymous client can verify user tokens
async verifyToken(token) {
  // Create fresh anonymous client for token verification
  const userClient = createClient(supabaseUrl, supabaseAnonKey);
  const { data: { user }, error } = await userClient.auth.getUser(token);
  // Success: Anonymous client supports ES256 user tokens
}
```

### 2. **Client Usage Matrix**

| Operation | Client Type | Key Used | Token Type | Signing Method |
|-----------|-------------|----------|------------|----------------|
| Verify user tokens | Anonymous | `SUPABASE_ANON_KEY` | User access token | ES256 ✅ |
| Admin operations | Service Role | `SUPABASE_SERVICE_ROLE_KEY` | Service token | HS256 ✅ |
| User operations | Anonymous | `SUPABASE_ANON_KEY` | User access token | ES256 ✅ |

### 3. **Code Changes**

**Before (Broken)**:
```javascript
class AuthService {
  constructor() {
    this.supabase = supabaseAdmin; // Service role client
    this.supabaseClient = supabaseClient; // Anonymous client
  }
  
  async verifyToken(token) {
    // Wrong: Using service role client for user token
    const { data: { user }, error } = await this.supabaseClient.auth.getUser(token);
    // Still fails because supabaseClient might be misconfigured
  }
}
```

**After (Fixed)**:
```javascript
async verifyToken(token) {
  // Create fresh anonymous client instance
  const { createClient } = await import('@supabase/supabase-js');
  const userClient = createClient(
    process.env.SUPABASE_URL, 
    process.env.SUPABASE_ANON_KEY
  );
  
  // Use anonymous client to verify user token
  const { data: { user }, error } = await userClient.auth.getUser(token);
  // Success: Correct client for correct token type
}
```

## Why This Happens

### Supabase Architecture
1. **Frontend** gets ES256-signed access tokens from Supabase Auth
2. **Frontend** sends these tokens to backend APIs
3. **Backend** must use anonymous client to verify ES256 tokens
4. **Service role client** is only for admin operations, not user token verification

### Common Misconception
Developers often think "service role = more powerful = better for verification", but:
- Service role clients are for **admin operations**
- Anonymous clients are for **user token verification**
- Each client type supports different token signing methods

## Security Considerations

### Anonymous Key Safety
- Anonymous key is **safe to use** in backend for user token verification
- It's the same key used in frontend applications
- It cannot perform admin operations
- It respects Row Level Security (RLS) policies

### Service Role Key Protection
- Service role key should **only be used** for admin operations
- Never expose service role key to frontend
- Use for user management, bypassing RLS, etc.
